# Internationalization (i18n)

Status: foundation (issue #141), written 2026-06-12. Stack:
`i18next` + `react-i18next` + `expo-localization` (Expo SDK 56 compatible),
layer in `apps/mobile/src/i18n/`.

## Principles

1. **Language files are product content, not just technology.** Every
   released locale is a complete, human-reviewed deliverable with its own
   quality bar ‚Äî like a feature, not like a config file.
2. **Claude/AI assistants do not translate languages as a side effect.** No
   task may "also quickly translate the app". A new language is its own
   issue: AI may draft, but a human reviews every string before the locale
   is released. **No unreviewed AI copy in released locales, ever.**
3. **No fake quality signals.** Incomplete or machine-generated English must
   never be visible. Until a locale is released, devices with that language
   see the German source ‚Äî honest and consistent beats half-translated.
4. **Source language is `de`.** The German strings are the authoritative
   copy; other locales derive from them.

## Released locales

| Locale | Status |
| --- | --- |
| `de` | source language, released |
| `en` | planned next target ‚Äî **not started, not released**; own issue when commissioned |

Releasing a locale means: complete language file that `satisfies Messages`,
human review done, code added to `RELEASED_LOCALES`
(`apps/mobile/src/i18n/locale.ts`) and the registry
(`apps/mobile/src/i18n/locales/registry.ts`). The parity test
(`i18n.test.ts`) fails on any key drift or empty value.

## Keys

- Stable semantic keys: `auth.signIn.title`, `devSession.start`,
  `claims.actions.archive`. Never natural-language keys
  (`"Bitte anmelden"` as key is forbidden).
- Renaming a key is a refactor; changing a value is a product copy change ‚Äî
  keep the two separate in PRs where practical.
- Group by feature area (`auth.*`, `devSession.*`, later `claims.*`,
  `groups.*`, ‚Ä¶), not by screen file name.

## Hardcoded-string rule

New or touched **UI copy** in `apps/mobile` goes through `t("‚Ä¶")` and
`locales/de.ts` ‚Äî no new hardcoded user-facing strings in JSX. The #142
extraction is **complete**: all screens (Profile auth cards #141; Claims,
Inbox, Overview; Groups, Record, tab bar, app shell), and the service-layer
result messages.

**Service-layer messages** (chunk 3): services and data adapters never hold
translated copy. They return a `ServiceMessage` ‚Äî a stable key into the
`service.*` block of the locale files plus params ‚Äî and only the UI
translates via `formatServiceMessage(t, message)`
(`apps/mobile/src/i18n/service-message.ts`, pure module, node-testable, no
expo/react imports; keys compile-checked at every `msg()` call site).
`{{detail}}` carries pass-through technical detail (server error text or a
nested service message). Error semantics unchanged.

**Deliberately NOT i18n keys** (final acceptance, #142):

- `mock-data/*` and the tab `ScreenContent` placeholder blocks ‚Äî demo data,
  including the Profile screen's mock-driven sections and demo fixture names
  ("Lokale Testgruppe", "Dev User").
- The supabase-local adapters' ScreenMock filling (`supabase-repositories.ts`
  detail/inbox titles, hints and status labels; `mock-repositories.ts` status
  labels): tied to the MVP-1A mock-shape prototype architecture and replaced
  wholesale when real group/activity navigation lands (1B scope) ‚Äî extracting
  them now would key copy that is scheduled for deletion.
- Developer diagnostics: the `supabase-local ¬∑ ‚Ä¶` status hints
  (`getStatusHint`/`getClaimsStatusHint`) and the unused
  `getSupabaseLocalConfigHint` env hint ‚Äî dev-mode debugging output.
- User-generated content (names, group titles, notes ‚Äî never translated) and
  amounts/currencies (separate concern).

**Rules**: dynamic label maps stay typed key maps (`TAB_LABEL_KEYS`,
`LIFECYCLE_KEYS`, ‚Ä¶), never string concatenation; interpolation uses
`{{n}}`/`{{name}}`/`{{date}}`/`{{detail}}` ‚Äî `{{count}}` only with real
`_one`/`_other` plural keys. English remains a separate future language
issue (own reviewed deliverable), not part of #142.

Device verification: the full i18n/ServiceMessage rendering passed an
emulator regression run on 2026-06-13 (#147, Android emulator + Expo Go,
both data modes ‚Äî see `demo-script.md`): no raw keys, interpolations and
translated service feedback correct.

## Locale resolution & fallback

Device languages come from `expo-localization` (`getLocales()`), resolution
is `resolveLocale()` in `locale.ts` (pure, unit-tested): first device
language that is a released locale wins, everything else falls back to `de`.
No runtime/network translation, no external translation services, resources
are bundled.

## Adding a new language (checklist)

1. Own issue (e.g. "Release locale en"). The locale is the deliverable.
2. Create `locales/<code>.ts` with the full `Messages` shape (`satisfies
   Messages` ‚Äî the compiler enforces parity).
3. Draft however you like (AI allowed), then **human review of every
   string** in product context.
4. Add the code to `RELEASED_LOCALES` + registry; tests enforce parity and
   non-empty values.
5. Manual QA pass on a device with that system language.
